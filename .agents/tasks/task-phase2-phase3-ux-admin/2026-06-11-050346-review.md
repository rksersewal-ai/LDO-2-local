# Upload progress, OCR status enrichment, and admin monitoring dashboard (v2)

The fix commit addresses all six v1 issues: progress simulation replaces the decorative jumps, drag-and-drop now simulates upload in-place without navigating away, the stale retry count is captured before clearing, all admin panels poll on 30-second intervals, the OcrQueueMonitor derives processing jobs from worker data, and uploads auto-dismiss after completion/failure. Watch for: interval leak on failed upload in DocumentIngestion (confirmed), and `getWorkers()` returning a static constant that defeats the polling intent in OcrQueueMonitor and WorkerStatusPanel (confirmed).

## High-level view

The progress simulation in `DocumentIngestion` uses a 300ms `setInterval` that advances progress by 5% increments until 90%. The interval is cleared on success but not on failure, so a rejected API call leaves a ticking interval attached to an orphaned upload ID. It becomes inert once the auto-dismiss removes the item from the map, but it still runs until the component unmounts — a minor resource leak that compounds in error-heavy sessions.

The drag-and-drop flow now simulates upload progress entirely in the floating panel (30% → 80% → complete on fixed timeouts) and no longer navigates away. This decouples the drop interaction from the ingest form entirely — a deliberate simplification that means dropped files are "uploaded" cosmetically but never actually ingested. The UX is no longer broken (phantom items that never complete), but the feature is now purely decorative: the user sees progress that represents nothing happening.

The admin polling fix adds identical `useEffect`/`setInterval` patterns across five components, all pointing at `AdminMetricsService.getMetrics()` or `getWorkers()`. The metrics path reads from localStorage so external updates are picked up. The workers path (`getWorkers()`) returns `DEFAULT_WORKERS` — a compile-time constant — so the panel will show the same four workers with identical heartbeat drift (growing stale by 30s per poll) regardless of any system state.

## Interval leak in DocumentIngestion progress simulation

The fix replaces the hardcoded 0→30→80→100 jumps with a smooth setInterval-based progression:

```typescript
const progressInterval = setInterval(() => {
  const current = UploadProgressService.getActive().find((item) => item.id === uploadId);
  if (current && current.progress < 90) {
    UploadProgressService.updateProgress(uploadId, current.progress + 5);
  }
}, 300);

const result = await apiClient.ingestDocument(formData);
clearInterval(progressInterval);
```

If the `await` rejects, execution jumps to the `catch` block. The `finally` block only calls `setIsSubmitting(false)`. The interval keeps firing — each tick calls `getActive().find(...)` against a map that still contains the item (with status "failed") until the 8-second auto-dismiss removes it. After that the find returns undefined and the interval becomes a no-op, but `setInterval` callbacks continue executing every 300ms until navigation or unmount. In a session where the user retries ingestion several times (failed network, bad file format), each failure stacks another leaked interval.

## Worker data never updates through polling

`AdminMetricsService.getWorkers()`:

```typescript
getWorkers(): WorkerInfo[] {
  return DEFAULT_WORKERS;
}
```

`DEFAULT_WORKERS` sets `lastHeartbeat` to timestamps computed at import time (e.g., `Date.now() - 30 * 1000`). The `WorkerStatusPanel` calls `formatHeartbeat()` which computes `Date.now() - new Date(iso).getTime()`. As wall-clock time advances, every worker's "last heartbeat" drifts further into the past. After 5 minutes on the page, the stale-worker alert will fire for workers that were initially "healthy" — not because anything changed, but because the static timestamps are aging against real time.

The `OcrQueueMonitor` derives its "Currently Processing" list from the same static array, so the processing list is now consistent with `WorkerStatusPanel` (fixing v1 issue #5) but still never reflects any actual state change. This is acceptable for a mock system if documented, but the polling infrastructure implies freshness that doesn't exist.

## Drag-and-drop is now a fire-and-forget animation

The v1 review flagged that dropped files navigated to the ingest page with no file data. The fix removes the navigation entirely and simulates progress on fixed timeouts:

```typescript
setTimeout(() => UploadProgressService.updateProgress(uploadId, 30), 500);
setTimeout(() => UploadProgressService.updateProgress(uploadId, 80), 1500);
setTimeout(() => UploadProgressService.completeUpload(uploadId), 2500);
```

The user sees files "upload" and complete in the floating panel, but no document is created, no metadata is collected, no ingest API is called. The toast says "queued for upload" which is misleading — nothing is queued anywhere. This is a strictly better UX than the v1 dead-end (no more phantom items), but it's worth noting that drag-and-drop is now a visual affordance with no backend action. Whether that's intentional depends on the product intent for this mock system.
