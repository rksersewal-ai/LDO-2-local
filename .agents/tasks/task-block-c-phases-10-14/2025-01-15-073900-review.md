# Security, reliability, and storage utility modules for EDMS frontend

Block C adds 16 new TypeScript modules (with co-located tests) across three domains: security hardening, operational reliability, and storage architecture. All modules are client-side utilities gated behind feature flags or usable standalone. A root Makefile provides developer/CI targets, and a production readiness doc summarizes the remediation effort.

**Watch for:** Hash-chain audit log uses a non-cryptographic hash that can be trivially forged by anyone with localStorage access (confirmed). The `maxEntries` truncation in AuditLogService breaks hash-chain integrity verification for any entry whose predecessor was pruned (confirmed). Several modules store security-sensitive state in localStorage with no encryption or integrity protection (confirmed).

## High-level view

The security modules provide five utilities covering audit logging, rate limiting, upload validation, session timeouts, and permission checking. The audit log chains entries with a deterministic hash, but the hash function is a 64-bit FNV-1a variant — fast and consistent but trivially reversible. Since the entire chain lives in localStorage, any script on the same origin can read, modify, or forge entries. The naming and documentation imply tamper detection that the implementation cannot deliver against a motivated actor.

The reliability modules are polling and tracking utilities: health check aggregation, backup record management, maintenance mode detection, pre-flight startup validation, and worker heartbeat liveness. All store state in localStorage and operate without network calls (except health check which takes injectable `checkFn` callbacks). The module naming (audit *integrity*, backup *verification*, worker *heartbeat*) implies operational infrastructure that is actually purely presentational — an operator cannot rely on `BackupTracker.isOverdue()` to know if backups are actually happening.

The storage modules model content-addressed path generation, directory layout constants, deduplication indexing, tiered storage policies, orphan file detection, and usage forecasting. The content hash function produces 64-char hex output by running four rounds of FNV-1a — deterministic and fast but not collision-resistant in the cryptographic sense.

The Makefile ties together test, lint, typecheck, and build commands, with placeholder targets for E2E and benchmarks. The `test` target hard-codes exclusions for two failing test files.

<summary>Issues (7)</summary>

1. **Audit log truncation breaks chain verification** — When `maxEntries` is exceeded, oldest entries are spliced out. Any subsequent `verifyChainIntegrity()` call will fail at entry 0 because its `previousHash` references a now-missing predecessor. Either store a "checkpoint hash" at truncation boundaries or verify only entries created after the last truncation.
2. **Non-cryptographic hash marketed as tamper detection** — `deterministicHash` is a 64-bit FNV-1a producing 16 hex chars. The doc says "tamper detection" but the hash provides no security against intentional modification. Rename or document the actual guarantee: consistency checking for accidental corruption, not adversarial tamper detection.
3. **localStorage as security state store** — Audit log, session state, backup records, and worker heartbeats all persist to localStorage. Any XSS on the same origin can read or overwrite this data silently.
4. **Rate limiter has no persistence** — `RateLimiter` is purely in-memory. A page reload resets all counters, making it ineffective against burst abuse patterns that reload the page.
5. **Content hash is not collision-resistant** — `hashContent` in `contentAddressedStorage.ts` produces 256 bits via four rounds of a 64-bit hash. This is structurally weaker than SHA-256 and should not be used for deduplication correctness guarantees in production; document this limitation or use SubtleCrypto when available.
6. **Makefile test exclusions are hardcoded** — The `test` target excludes `ThemeContext.test.tsx` and `useOverloadProtection.test.ts` by name. If those tests are fixed, someone must remember to update the Makefile. A config-level skip would be more maintainable.
7. **Upload validation regex for event handlers is overly broad** — The pattern `/on\w+\s*=/i` in `SCRIPT_PATTERNS` will flag benign attributes like `one=` in non-HTML content. For an EDMS handling engineering documents (likely SVGs with metadata), this creates a non-trivial false positive surface.

## Audit log hash chain and truncation conflict

The `AuditLogService` maintains an append-only log where each entry's `currentHash` is computed from its own fields plus the `previousHash` from the preceding entry. The verification function walks the entire array checking these links. However, `logOperation` enforces a `maxEntries` cap by splicing oldest entries:

```typescript
if (entries.length > this.maxEntries) {
  entries.splice(0, entries.length - this.maxEntries);
}
```

After truncation, entry[0] in the stored array will have a `previousHash` pointing to an entry that no longer exists. The next call to `verifyChainIntegrity()` will fail at index 0 because `expectedPreviousHash` is `GENESIS_HASH` (empty string) but the actual stored entry references the hash of a now-deleted predecessor. This makes the integrity check unusable for any log that has exceeded its capacity — exactly the long-running logs where integrity matters most.

## Hash function security posture

The `deterministicHash` function in `AuditLogService.ts` uses a dual FNV-1a variant producing a 53-bit combined value (via `4294967296 * (2097151 & h2) + (h1 >>> 0)`) represented as a 16-char hex string. The docstring calls this "tamper detection" and the overall service doc says "cryptographically linked." The hash has no preimage resistance and trivial collision generation. An attacker (or even a curious user in DevTools) can recompute valid hashes for forged entries.

The `hashContent` function in `contentAddressedStorage.ts` takes a different approach — running four rounds with offset seeds to produce 64 hex chars (256 bits). Better distribution, but still lacks cryptographic guarantees. For content addressing where the goal is deduplication, accidental collisions are unlikely but intentional ones are trivial to construct.

## Rate limiter lacks durability

`RateLimiter` stores request timestamps in a `Map` instance. A page refresh, tab close, or navigation resets all state. The module's API surface (`isLimited`, `getRemainingRequests`) implies security enforcement that evaporates on reload. Server-side enforcement remains necessary for any rate limiting that actually matters.

## Upload validation false positive risk

The `SCRIPT_PATTERNS` array includes `/on\w+\s*=/i` which matches any `on` + word-characters + `=` pattern. In SVG content, legitimate attributes can trigger this (`<metadata one="1">` would false-positive, though actual event handlers like `ontransitionend=` would correctly match). For an EDMS handling engineering documents with SVG metadata, this creates a non-trivial false positive surface. A tighter pattern like `/on(click|load|error|mouse\w+|key\w+|focus|blur)\s*=/i` would reduce false positives while catching real threats.

## Makefile and CI integration

The Makefile's `test` target hardcodes two test exclusions:

```makefile
test:
    npx vitest run && cd artifacts/edms && npx vitest run --exclude 'src/contexts/ThemeContext.test.tsx' --exclude 'src/hooks/useOverloadProtection.test.ts'
```

Placeholder targets (`test-e2e`, `benchmark-search`, `benchmark-ocr`, `backup-test`) echo messages rather than failing, so CI won't catch when these are expected to be implemented but aren't yet. Making placeholders exit non-zero with a "not yet implemented" message would surface the gap earlier.

## Storage tier policy's `Date.now()` coupling

`storageTierPolicy.ts` uses `new Date()` inside `getDaysSince()` without any dependency injection for the current time. While `SessionManager` accepts a `now` parameter for testability, `TierPolicy` does not. Tests for tier classification must manipulate `lastAccessedAt` dates relative to the real clock, making them time-sensitive — a file accessed "7 days ago" in a test might cross the threshold boundary depending on the exact time of day the test runs.
