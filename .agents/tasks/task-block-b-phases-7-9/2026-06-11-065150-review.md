# Search, performance instrumentation, and crash protection for EDMS

Block B adds three layers of infrastructure beneath the existing EDMS application: a PostgreSQL full-text search abstraction (Phase 7), client-side performance tooling (Phase 8), and a resilience stack with file validation, retry logic, circuit breaking, and structured error codes (Phase 9). All 20 source files are additive with no modifications to pre-existing business logic. The implementation is purely frontend/config-layer; no backend API endpoints are introduced, so the search and resilience primitives exist as type-safe descriptors awaiting server-side wiring.

**Watch for:** The search term sanitizer strips a narrow set of characters, leaving injection vectors open for query descriptors that reach a real database (confirmed). The global module-level `Map` in `useRequestDeduplication` leaks between test suites and would persist across micro-frontend boundaries in production (confirmed). Four source files (`imageLazyLoading`, `bundleSizeBudget`, `searchBenchmarkConfig`, `searchFiltersIntegration`) ship without any unit tests (confirmed).

## High-level view

The search layer (Phase 7) defines a query descriptor model rather than raw SQL. `searchQueryBuilder` composes tsvector clauses, GIN filter clauses, sort, and pagination into a serializable object. `SearchCacheService` provides an LRU+TTL localStorage cache that will bridge the gap until Redis is introduced. The engineering filters integration ties domain-specific filters (drawing number patterns, revision ranges, confidence thresholds) into the generic query builder.

Performance instrumentation (Phase 8) is a collection of standalone utilities: a request timer with percentile aggregation, an IntersectionObserver factory for lazy images, a deduplication hook with both per-component and global request maps, and a render-time monitor built on `performance.mark`/`measure`. Bundle size budgets are defined in code for CI enforcement.

The crash protection layer (Phase 9) covers the path from upload validation (file size, MIME, magic bytes) through processing safety (dimension limits, request timeouts, exponential backoff) to failure visibility (dead letter queue, circuit breaker, structured error codes). The `safeCleanup` scope provides LIFO resource teardown with isolated error handling.

1. **Search term sanitizer too narrow** — `sanitizeSearchTerm` only strips `\'"`;` but leaves `--`, `/*`, `(`, `)`, `&`, `|` intact. If descriptors ever reach a SQL layer without parameterized queries, these characters enable injection. Expand the strip set or document that backend parameterization is mandatory.

## Search query descriptor and sanitization gap

`sanitizeSearchTerm` strips only `\`, `'`, `"`, `` ` ``, and `;`. PostgreSQL tsquery operators like `&`, `|`, `!`, `(`, `)` pass through unchanged, as do SQL comment sequences `--` and `/*`. If the descriptor's `searchTerm` is ever interpolated into a `to_tsquery()` call without further escaping, it will produce parse errors or unintended query semantics.

```typescript
export function sanitizeSearchTerm(term: string): string {
  return term
    .replace(/[\\'"`;]/g, "")  // misses & | ! ( ) -- /*
    .replace(/\s+/g, " ")
    .trim();
}
```

The risk is mitigated by the descriptor pattern (the backend should parameterize), but the sanitizer's name implies safety it doesn't fully provide. Either expand it to strip tsquery-special characters or rename it to `normalizeSearchTerm` and document that backend parameterization is the security boundary.

## Global module-level state in request deduplication

`useRequestDeduplication` exports both a React hook (instance-scoped via `useRef`) and a standalone `deduplicateRequest` function backed by a module-level `globalInflightMap`. This global Map never expires entries if a promise hangs indefinitely — the `.finally()` handler only fires on settlement. A hung fetch means the Map grows without bound for that key, and subsequent callers receive a promise that never resolves.

The hook's instance-scoped map has the same hang risk but at least clears on unmount. The global map has no lifecycle owner. In test suites this causes cross-test contamination (the exported `clearGlobalInflightMap` exists for testing but requires discipline to call).

## localStorage saturation strategy in SearchCacheService

The cache stores each entry as a separate localStorage key (`search_cache_<hash>`) plus a stats key and an index key. When `saveEntry` catches a quota error, it calls `evictLRU()` once and retries. If the cache is competing with DLQ entries, API metrics, and other localStorage consumers for space, a single eviction may not reclaim enough. The failure path after the retry is silent — the entry simply isn't cached, with no signal to the caller.

## Full-jitter backoff floor at zero

`calculateBackoffDelay` applies full jitter: `Math.floor(Math.random() * cappedDelay)`. When `Math.random()` returns a value near 0, the delay is 0ms, meaning the retry fires immediately. In a client context with a single caller, a zero-delay retry is a wasted attempt against a failing service. Adding a minimum floor (e.g., `Math.max(100, jitter)`) would prevent no-op retries without losing the thundering-herd benefit of jitter.

## Performance mark accumulation

`usePerformanceMonitor` creates a new `performance.mark()` on every render (the call is at the top of the hook body, outside any effect). Cleanup via `clearPerformanceEntries` only runs on unmount. A component that re-renders 200 times before unmounting accumulates 200 mark entries and 200 measure entries in the browser's Performance timeline. The `MAX_RENDER_HISTORY = 100` cap applies only to the internal `durationsRef` array, not to the Performance API entries.

In development with React StrictMode (which double-renders), this doubles the accumulation rate. The cleanup iterates all marks/measures by type and filters by component name prefix — at scale this is O(n) over all performance entries in the page.

## Circuit breaker overall status is binary

`calculateOverallStatus` returns "down" if *any* registered service is down. In a system with optional services (e.g., cache is degradable), a single non-critical service failure marks the entire system as "down". The service doesn't distinguish critical vs optional services, so consumers of `getSystemStatus().overall` may over-react to cache or search outages that shouldn't block core document operations.

## Test coverage gaps

Four Phase 7-8 source files have no corresponding test file:

- `imageLazyLoading.ts` — IntersectionObserver factory, srcSet builder, priority calculator
- `bundleSizeBudget.ts` — budget check logic, report formatting
- `searchBenchmarkConfig.ts` — benchmark scenario definitions, utility functions
- `searchFiltersIntegration.ts` — engineering filter composition, revision range validation, confidence mapping

The search filters integration module contains non-trivial validation logic (revision range ordering, confidence level mapping) that would benefit from unit tests to prevent regression.
