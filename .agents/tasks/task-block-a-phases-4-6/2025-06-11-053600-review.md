# Engineering Document Intelligence, Tiled OCR Pipeline, and Large Drawing Viewer

Three features delivered as a cohesive vertical slice: regex-based metadata extraction from OCR text (Phase 4), a localStorage-backed tiled OCR pipeline service with types and polling hook (Phase 5), and a pan/zoom/rotate drawing viewer with OCR overlay composition (Phase 6). The implementation is cleanly layered: pure utility functions at the bottom, a service layer in the middle, React components and hooks at the top. Feature flags gate all runtime behavior via `VITE_FF_*` environment variables.

**Watch for:** React Rules of Hooks violations in two viewer components that early-return before calling hooks (confirmed). The PL number extractor matches any standalone 8-digit number, which will produce false positives on dates, phone fragments, and other numeric data in real OCR output (confirmed). The polling hook does not reset stale state when `jobId` changes mid-lifecycle (likely).

## High-level view

The document intelligence utilities are pure functions with no side effects — drawing number, revision, and PL number extractors paired with validation rules and confidence scoring. They form the extraction foundation for OCR post-processing but the PL number regex is overly broad, treating any 8-digit boundary-delimited number as a match.

The OCR pipeline layer introduces a YAML-mirrored config, rich type system with runtime type guards, and a localStorage-backed mock service. The service is feature-flag-gated at every method call, which is defensive but means the flag is evaluated on every interaction rather than once at construction. The `useTiledOcrPolling` hook manages interval-based polling with proper cleanup, though it doesn't handle the case where `jobId` changes while polling is active.

The drawing viewer composes pan/zoom/rotate interactions, keyboard shortcuts, and an OCR text overlay into a single `TiledDrawingViewer` component. Both `TiledDrawingViewer` and `LargeDrawingViewerPage` place conditional returns before React hooks, which violates the Rules of Hooks and will fail under React's strict mode or a linter pass. The OCR overlay positions regions as percentage-based absolute elements and color-codes them by confidence — a clean approach for visual inspection of extraction quality.

## Rules of Hooks violations in viewer components

Both `TiledDrawingViewer` and `LargeDrawingViewerPage` guard against a disabled feature flag by returning early at the top of the function body:

```tsx
// TiledDrawingViewer.tsx line 49
if (!isFeatureEnabled("LARGE_DRAWING_VIEWER")) {
  return null;
}
// ... then useState, useCallback, useEffect follow
```

```tsx
// LargeDrawingViewerPage.tsx line 36
if (!isFeatureEnabled("LARGE_DRAWING_VIEWER")) {
  return (...);
}
// ... then useMemo calls follow
```

React requires that hooks execute in the same order on every render. A conditional return before hooks means on one render the hooks run and on another they don't. Since `featureFlags` is frozen at module load, the flag won't actually flip between renders in the current implementation — but this is still a violation that React's development mode and ESLint's `react-hooks/rules-of-hooks` rule will flag. The fix is to move the early return below all hook calls, or extract a wrapper component that checks the flag and conditionally renders the inner (hook-using) component.

## PL number extraction: false-positive surface

`extractPLNumbers` uses `\bPL[-\s]?(\d{8})\b|\b(\d{8})\b` — the second alternation matches any standalone 8-digit number. In real OCR text from engineering drawings, 8-digit sequences appear frequently: dates (20240315), warranty codes, serial numbers, dimension sequences. The test suite only validates the happy path and the 7-digit/9-digit exclusion, not false-positive rejection.

Without a contextual heuristic (proximity to "PL" label, position in title block region, or semantic filtering), this extractor will inject noise into metadata. The first alternation (`PL[-\s]?\d{8}`) is well-targeted; the standalone `\d{8}` fallback is the issue.

## Polling hook lifecycle gap

`useTiledOcrPolling` builds `fetchJob` with `useCallback([jobId])` and `startPolling` with `useCallback([fetchJob, intervalMs])`. When `startPolling` is invoked, it creates an interval that calls `fetchJob`. If `jobId` changes while polling is active, `fetchJob` gets a new identity (since `jobId` is in its dependency array), but the running `setInterval` still holds a reference to the old `fetchJob` closure. The interval won't pick up the new job ID until the consumer explicitly stops and restarts polling.

A `useEffect` that restarts the interval whenever `jobId` changes (while `isPolling` is true) would close this gap.

## Feature flag gating strategy

The system uses two distinct flags: `TILED_OCR` gates the service and hook layer, while `LARGE_DRAWING_VIEWER` gates the viewer components. This separation means a deployment could enable the viewer without the OCR service, in which case `LargeDrawingViewerPage` would render the viewer but `TiledOcrService.getJobsForDocument` would return `[]` (no tile status panel shown). The graceful degradation is fine, but the relationship isn't documented — a consumer might expect both flags to be required together.

## OCR overlay coordinate mapping

The overlay converts tile pixel coordinates to percentages relative to the maximum bounding box of all tiles:

```tsx
const maxX = Math.max(...activeJob.tiles.map((t) => t.coordinates.x + t.coordinates.width));
```

This assumes tiles form a contiguous rectangle starting at (0, 0). If the tiling backend ever produces non-zero-origin tiles or non-rectangular layouts, the overlay positions will be incorrect. For the current mock data this works, but the assumption should be documented or asserted.

## Test coverage shape

The extraction utilities (Phase 4) have 94 tests covering happy paths, edge cases, and empty/null inputs. The OCR service has 46 tests covering flag gating, CRUD operations, and persistence. Type guards are thoroughly tested.

Not tested: the `useTiledOcrPolling` hook has no test file. The viewer components (`TiledDrawingViewer`, `OcrTextOverlay`, `DrawingViewerControls`, `TilePyramidStatus`, `LargeDrawingViewerPage`) have no component tests. Keyboard shortcut behavior, drag interaction, and wheel zoom are exercised only in production renders. The `engineeringSearchFilters` module has no dedicated test file — `buildFilterQuery`, `getFilterTemplate`, and `confidenceLevelToFilter` are untested.
