# Gap closure plan, typecheck fixes, a11y fixes, and feature-flag scaffolding

This change introduces a comprehensive gap closure plan (677 lines) with 8 architecture decision records, fixes all 16 TypeScript errors across test files and the WorkLedger component, resolves 7 lint errors related to label accessibility and static element interactivity, adds a feature flags utility for gating future features behind environment variables, and provides an OCR pipeline configuration file.

The approach is documentation-first: define the path to production readiness through ADRs and a structured plan, then make the codebase pass its own CI gates (typecheck, lint reduction). The feature flags utility and OCR config are forward-looking scaffolding referenced by the plan.

Watch for: custom test matchers replacing `@testing-library/jest-dom` are incomplete (only 2 of ~20 matchers implemented) — tests relying on other matchers like `toHaveAttribute` or `toHaveTextContent` will fail silently or crash (**confirmed**); the `Select` component receives an `id` prop for label association but whether it forwards it to the underlying `<select>` element is unverified (**likely**); the `featureFlags` object is frozen at module load time making it untestable without module-reload gymnastics (**confirmed**).

## High-level view

The typecheck fix strategy is pragmatic: exclude test files from the production `tsconfig.json` and inline custom matchers in the test setup rather than fixing the `@testing-library/jest-dom` type incompatibility. This eliminates all 16 errors but hides future type drift in test files from the main typecheck pass — test-only type errors won't surface until tests run.

The accessibility fixes wire `htmlFor`/`id` pairs for form controls and add `role="button"` with keyboard handlers to the file drop zone. The `PLNumberSelect` component threads `id` to the underlying button correctly, but `Select` also receives `id` and its forwarding behavior isn't visible in this diff — a potential broken label association for two form fields.

The feature flags module provides a read-once pattern with Vite's `import.meta.env` and frozen output. Flags are effectively compile-time constants after build, which is acceptable for LAN deployment but makes unit testing flag-dependent code paths awkward without module-reload isolation.

The OCR pipeline config's `skip_tile` fallback delivers partial results rather than aborting on tile failure — appropriate for engineering contexts where a missing region can be manually reviewed. The shared `temp_directory` path is a collision risk in multi-worker deployments.

## TypeScript error resolution via tsconfig exclusion

The 16 typecheck errors fell into two categories: test files importing `@testing-library/jest-dom` (which has known type conflicts with Vitest's matcher interface), and `WorkLedger.tsx` passing only one argument to `createPortal` (missing the `container` argument).

The test-file errors are resolved by excluding test patterns from the production `tsconfig.json`. This means `pnpm run typecheck` no longer checks test files at all — type drift in tests will only surface when Vitest actually runs them.

The `@testing-library/jest-dom` import is replaced with custom matchers in `setup.ts`. Only two matchers are implemented:

```typescript
expect.extend({
  toBeInTheDocument(received: unknown) { /* checks document.contains() */ },
  toHaveClass(received: unknown, ...classNames: string[]) { /* checks classList */ },
});
```

The existing test file (`ErrorBoundary.test.tsx`) only uses `toBeInTheDocument`, so it passes. But any future test using `toHaveAttribute`, `toHaveTextContent`, `toBeVisible`, `toHaveStyle`, or other jest-dom matchers will get a runtime error (`not a function`) rather than a type error at check time. The type declarations in `setup.ts` only declare two matchers, so TypeScript will catch *some* missing ones in test files (if they're ever type-checked), but the `expect.extend` registry is the real bottleneck.

## Accessibility: label association gap in Select

The `DocumentIngestion.tsx` form adds `htmlFor`/`id` pairs for each form control. For `<Input>` components this works directly. For `<PLNumberSelect>`, the new `id` prop threads to the combobox trigger button:

```tsx
<Button type="button" variant="outline" role="combobox" id={id} ...>
```

However, `<Select>` receives `id="doc-type-select"` and `id="doc-category-select"` and its implementation isn't shown in this diff. If `Select` is a styled wrapper that doesn't forward `id` to the native `<select>`, the label click won't move focus — silently broken accessibility for two required fields.

## Feature flags: type/array duplication and testability

Adding a new flag requires updating both the `FeatureFlagKey` type union and the `FLAG_KEYS` array independently. Nothing enforces they stay in sync — a key added to the union but missing from the array will type-check but never be read from the environment.

The frozen-at-import pattern means testing code paths behind different flag values requires `vi.resetModules()` and dynamic `import()` between assertions. An alternative would be exposing a `createFlags(env)` factory that tests can call with controlled input, while the module-level `featureFlags` remains the production entry point.

## OCR pipeline: temp directory collision

The `cleanup_policy: on_success` leaves temporary files around on failure, which aids debugging but accumulates disk usage. The fixed `temp_directory: /tmp/ocr_tiles` path means concurrent jobs from multiple workers will write tiles to the same directory. Unless the consuming service namespaces by job ID within this path, tile files from different documents could overwrite each other or be erroneously cleaned up by a succeeding job while a failing job's files are still needed for debugging.
