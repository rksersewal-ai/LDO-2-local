# Upload progress, OCR status enrichment, and admin monitoring dashboard

Two feature commits add user-facing upload tracking, richer OCR status display, drag-and-drop ingestion, recent-document history, saved filter presets, and a suite of admin monitoring components (OCR queue, worker status, storage, failed jobs). The implementation relies on localStorage-backed services with a custom event bus for cross-component communication (uploads) and static mock data for the admin panels. Watch for: progress tracking that never reaches real server progress (confirmed), admin metrics that are entirely client-side mocks with no refresh mechanism (confirmed), and a stale-data race in the `FailedJobsPanel` retry flow (likely).

## High-level view

Upload progress uses a module-scoped `Map` in `UploadProgressService` with `CustomEvent` dispatch to a floating panel in `AppLayout`. The progress values in `DocumentIngestion` are hard-coded (0 → 30 → 80 → 100), making the progress bar decorative rather than informative.

Drag-and-drop on `DocumentHub` registers uploads via `UploadProgressService.startUpload` and navigates to the ingest page, but never wires those "queued" uploads to actual form state in `DocumentIngestion`, so the user lands on an empty form with phantom progress items in the floating panel that will never complete.

The admin monitoring suite (`SystemHealthSummary`, `OcrQueueMonitor`, `WorkerStatusPanel`, `StorageUsageIndicator`, `FailedJobsPanel`) renders entirely from `AdminMetricsService` defaults stored in localStorage. None of these components poll or subscribe to real-time updates, so the dashboard is a static snapshot that only changes when the user manually retries a job.

`OcrQueueMonitor` contains a hardcoded `processingJobs` array inside the component body, duplicating concepts already expressed in `AdminMetricsService.getWorkers()`. The "Currently Processing" list never reflects reality even within the mock system.

1. **Decorative progress bar** — `DocumentIngestion` hard-codes progress at 30/80/100 without correlating to actual upload byte progress. Either wire `XMLHttpRequest`/fetch progress events or remove the percentage display to avoid misleading users. (confirmed)
2. **Drag-and-drop ↔ ingest disconnect** — Files dropped on `DocumentHub` create phantom upload items and navigate to `/documents/ingest`, but the ingest form has no awareness of the dropped files. Pass file references through state or a shared service so the form pre-populates. (confirmed)
3. **Stale retry count in toast** — `FailedJobsPanel.handleRetryAll` captures `jobs.length` from the pre-retry state but the toast fires after `setJobs([])`, so the count could drift if a re-render intervenes. Capture the count before clearing. (likely)
4. **No polling or subscription in admin panels** — All admin components call service methods once (either in `useState` initializer or inline render). 5. **Hardcoded processing jobs in OcrQueueMonitor** — The "Currently Processing" list is a static array inside the component, not derived from the service layer. This creates a maintenance gap and a data inconsistency with `WorkerStatusPanel`. (confirmed)
6. **Upload items never auto-cleared** — Completed/failed uploads persist in the floating panel until the user clicks dismiss. For batch drag-and-drop of many files this panel will grow unbounded within a session. Add auto-dismiss after a timeout. (confirmed)

## Upload progress tracking and the phantom-file gap

In `DocumentIngestion.tsx`, the integration calls `startUpload` → `updateProgress(30)` → API call → `updateProgress(80)` → `completeUpload`. These fixed waypoints don't reflect actual network progress. The browser `fetch` API doesn't expose upload progress natively, but `XMLHttpRequest` does, and the existing `apiClient.ingestDocument` wraps fetch. So either the service needs an XHR path for progress, or the UI should display an indeterminate state rather than a percentage that jumps from 0 to 30 to 80.

The drag-and-drop flow in `DocumentHub` has a more fundamental issue. When files are dropped:

```typescript
UploadProgressService.startUpload(`drop_${Date.now()}_${i}`, files[i].name);
// ... toast, then:
setTimeout(() => navigate("/documents/ingest"), 900);
```

The `File` objects are never stored anywhere accessible to `DocumentIngestion`. The user arrives at the ingest page with an empty file picker while the floating panel shows "uploading" items that will never complete—no code ever calls `completeUpload` or `failUpload` on those IDs. This is a dead-end UX path.

## Admin metrics: static mocks with no refresh

Every consumer reads data synchronously during render:

- `SystemHealthSummary`: calls `getMetrics()` in a `const` outside any effect
- `OcrQueueMonitor`: same pattern
- `StorageUsageIndicator`: same
- `WorkerStatusPanel`: calls `getWorkers()` which returns a hardcoded constant (`DEFAULT_WORKERS`)

There is no `setInterval`, no event listener, no SSE/WebSocket integration. The panels render once and never update unless the component remounts. For a monitoring dashboard, an admin staring at the page will see the same numbers indefinitely. `FailedJobsPanel` is the one exception: its retry handlers call `setJobs(updated)`, so user-initiated mutations reflect locally.

## Stale closure in FailedJobsPanel.handleRetryAll

```typescript
const handleRetryAll = () => {
  const updated = AdminMetricsService.retryAllFailed();
  setJobs(updated);
  toast.success("All failed jobs queued", {
    description: `${jobs.length} job(s) re-added to the processing queue.`,
  });
};
```

`jobs.length` closes over the state value at the time the handler was created. If a job was individually retried between renders (shrinking the array), the bulk-retry toast will report the stale count. Capture `const count = jobs.length` before the service call.

## StorageUsageIndicator hardcoded breakdown

The "Breakdown" section in `StorageUsageIndicator` contains:

```typescript
const breakdown = [
  { label: "Documents", value: "89.2 GB", percent: 62 },
  { label: "OCR Outputs", value: "34.1 GB", percent: 24 },
  { label: "Backups", value: "19.4 GB", percent: 14 },
];
```

These numbers don't derive from `AdminMetricsService.getMetrics()`. They happen to sum to 142.7 (matching the default `storageUsedGB`), but if localStorage holds different values the breakdown will diverge from the header total. The breakdown should be served by the metrics service or computed proportionally.
